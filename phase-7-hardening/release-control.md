## 3) Definitions
Release ID:
- Immutable identifier for one FE/BE pair and one deployment event.
Build Pair:
- Exact FE commit SHA + exact BE commit SHA approved together.
Parity Evidence:
- Machine-verifiable proof that deployed FE/BE match approved pair.
Receipt Bundle:
- Mandatory artifacts attached before and after release.
Train Release:
- Scheduled release under normal controls.
Hotfix Release:
- Emergency release for active production incident.
Release ID format:
`ATH-RLS-YYYYMMDD-<train|hotfix>-NN`
Example:
`ATH-RLS-20260305-train-01`
Evidence path format:
`simulation/athelon/evidence/releases/<release-id>/`

## 4) FE/BE build parity policy
Policy statement:
- FE and BE ship as one approved unit.
- FE-only or BE-only production promotion is prohibited.
- Pairing is immutable after GO.
- Any mismatch is automatic FAIL.
### 4.1 Immutable hash pairing controls
P-01 FE commit recorded as full 40-char SHA.
P-02 BE commit recorded as full 40-char SHA.
P-03 FE artifact hash recorded as sha256.
P-04 BE package hash recorded as sha256.
P-05 Pairing record generated by CI, never manual.
P-06 Pairing record contains FE SHA, BE SHA, FE hash, BE hash.
P-07 Pairing record signed by release bot key.
P-08 Pairing record stored in immutable path.
P-09 Pairing record cannot be edited post-GO.
P-10 Any post-GO change requires new Release ID.
P-11 Runtime FE build metadata includes Release ID.
P-12 Runtime BE metadata includes Release ID.
P-13 Runtime metadata endpoint exposes commit SHA and Release ID.
P-14 Pre-release script validates runtime metadata endpoints in target env.
P-15 Deployed hashes are recomputed from served artifacts and compared.
### 4.2 Release ID controls
P-16 Release ID created before build starts.
P-17 Release ID linked to one build pair only.
P-18 Rebuild with same code still gets a new Release ID.
P-19 Approvals, checklists, and evidence must all reference same Release ID.
P-20 Missing Release ID at any step is FAIL.
### 4.3 Artifact retention controls
P-21 Retain FE manifest 18 months.
P-22 Retain BE manifest 18 months.
P-23 Retain parity pairing record 24 months.
P-24 Retain approvals/checklists 24 months.
P-25 Retain screenshots/videos 12 months.
P-26 Retention starts at GO timestamp.
P-27 Early deletion is prohibited.
P-28 Store evidence in immutable-capable object store.
P-29 Nightly backup of release evidence index required.
P-30 Quarterly restore drill required.
### 4.4 Parity acceptance criteria
All must PASS:
- FE deployed commit == approved FE commit.
- BE deployed commit == approved BE commit.
- FE deployed hash == FE manifest hash.
- BE deployed hash == BE manifest hash.
- FE runtime Release ID == release record Release ID.
- BE runtime Release ID == release record Release ID.
- Pairing record signature verifies.
- No untracked deployment between approval and GO.
Any FAIL = NO-GO or rollback decision per section 9.

## 5) Release train controls
### 5.1 Cadence
Primary train: Thursday.
Optional stabilization train: Monday.
No unscheduled train release outside hotfix policy.
### 5.2 Freeze windows
Code freeze: T-24h.
Approval freeze: T-2h.
Evidence freeze: T-30m.
No code or config promotion after freeze without restart of checklist.
### 5.3 Who can cut release
Only Release Controller from protected branch.
MFA session required.
Operator identity logged.
Terminal/session log retained as evidence.
### 5.4 Required approvals
Required for train release:
- FE owner approval.
- BE owner approval.
- QA approval.
- Compliance approval when integrity/compliance-impacting change exists.
- Release Controller final approval.
Conditional:
- Product approval for user-facing workflow change.
- Executive approval only for explicit risk acceptance exception.
Rules:
- Approvals expire after 24h.
- Any code change invalidates approvals.
- Approvals must be machine-readable and tied to Release ID.
- Copied-forward approvals are invalid.
### 5.5 Separation of duties
No single person may:
- Implement all affected code,
- Approve all gates,
- And execute release alone.
At least two humans must sign and one must be non-implementer for impacted layer.
### 5.6 Emergency hotfix path
Entry criteria:
- Active SEV-1 or SEV-2 incident,
- No safe workaround,
- Clear customer/compliance impact.
Minimum approvals:
- Impacted layer owner,
- QA quick validation,
- Release Controller execution.
Hotfix constraints:
- Scope limited to incident fix.
- No opportunistic feature bundling.
- Branch includes incident ID.
- Release ID uses `hotfix` segment.
- Full parity evidence still mandatory.
- Retro within 24h mandatory.
Rollback requirement:
- Named rollback target required before deploy.
- Rollback command and owner documented.

## 6) Pre-release checklist (binary PASS/FAIL)
Checklist owner: Release Controller.
Witnesses: QA lead and compliance when required.
No WARN state.
No PARTIAL state.
### 6.1 Gate Set A — source/build integrity
A1 Protected branch clean.
PASS: no unreviewed commit.
FAIL: unknown commit or direct push.
A2 Release ID present.
PASS: ID exists and unique.
FAIL: missing/duplicate.
A3 FE SHA pinned.
PASS: full 40-char SHA recorded.
FAIL: short/moving ref.
A4 BE SHA pinned.
PASS: full 40-char SHA recorded.
FAIL: short/moving ref.
A5 FE CI required jobs green.
PASS: all required checks pass.
FAIL: any required red.
A6 BE CI required jobs green.
PASS: all required checks pass.
FAIL: any required red.
A7 Pairing record generated.
PASS: `parity/pairing.json` exists and signed.
FAIL: missing/invalid.
A8 FE/BE hashes recorded.
PASS: both sha256 values exist.
FAIL: missing either.
A9 Hash recompute checks.
PASS: recompute == recorded for FE and BE.
FAIL: mismatch.
A10 No bypassed checks.
PASS: no manual override on required checks.
FAIL: override detected.
### 6.2 Gate Set B — deployment readiness
B1 Target environment health.
PASS: platform and dependencies green.
FAIL: active degradation.
B2 Schema compatibility.
PASS: migration path validated.
FAIL: unsafe/unvalidated migration.
B3 Signature auth route readiness.
PASS: non-404 route + event creation test passes.
FAIL: route/event failure.
B4 Audit write readiness.
PASS: smoke action writes expected audit event.
FAIL: no write/delayed write.
B5 Org isolation probe.
PASS: cross-org access denied.
FAIL: leak/exposure.
B6 Incident readiness.
PASS: on-call chain confirms response in <5 min.
FAIL: no reachable responder.
B7 Rollback readiness.
PASS: last-known-good release and commands verified.
FAIL: rollback uncertain.
B8 Evidence storage readiness.
PASS: write/read to evidence path successful.
FAIL: storage failure.
### 6.3 Gate Set C — launch-blocker regression checks
C1 PDF export positive path.
PASS: valid export generated and opened.
FAIL: absent/broken output.
C2 PDF export negative path.
PASS: explicit user-visible failure and audit trace.
FAIL: silent failure or false success.
C3 IA/A&P separation.
PASS: fields distinct in IA-required sign flow.
FAIL: conflated values.
C4 Null-IA hard guard.
PASS: IA-required signing blocked if IA missing.
FAIL: bypass possible.
C5 Canonical hash parity.
PASS: independent recompute matches stored hash.
FAIL: mismatch.
C6 QCM live convergence.
PASS: second session updates without refresh.
FAIL: refresh required.
C7 Auth event consume semantics.
PASS: fresh event required and consumed once.
FAIL: reuse or missing consume.
C8 RTS guard-order regression.
PASS: enforced guard order matches approved spec.
FAIL: order drift.
### 6.4 Pre-release decision rule
GO-CANDIDATE requires all A/B/C gates PASS.
Any FAIL = NO-GO.
NO-GO actions:
- Open blocking incident.
- Assign owner.
- Record ETA or unknown.
- Re-run full checklist after fix.

## 7) Post-release checklist (binary PASS/FAIL)
Post-release window target: 30 minutes.
Owner: Release Controller.
Witness: QA lead.
### 7.1 Gate Set D — deployed parity confirmation
D1 FE runtime commit check.
PASS: runtime FE SHA == approved FE SHA.
FAIL: mismatch.
D2 BE runtime commit check.
PASS: runtime BE SHA == approved BE SHA.
FAIL: mismatch.
D3 FE runtime Release ID check.
PASS: runtime FE Release ID == record.
FAIL: mismatch/empty.
D4 BE runtime Release ID check.
PASS: runtime BE Release ID == record.
FAIL: mismatch/empty.
D5 FE deployed hash check.
PASS: served FE hash == manifest hash.
FAIL: mismatch.
D6 BE deployed hash check.
PASS: deployed BE hash == manifest hash.
FAIL: mismatch.
D7 Parity evidence upload completeness.
PASS: all required parity receipts present.
FAIL: missing receipt.
### 7.2 Gate Set E — functional smoke confirmation
E1 Sign-in and org claims.
PASS: expected org/role claims resolved.
FAIL: claim error.
E2 Work order create/open.
PASS: flow completes.
FAIL: unexpected block.
E3 Signing flow with auth.
PASS: fresh auth required and signature succeeds.
FAIL: bypass/failure.
E4 QCM review path.
PASS: review writes and dashboard converges.
FAIL: stale state.
E5 Export + audit pairing.
PASS: export succeeds and `record_exported` visible.
FAIL: missing export or missing audit.
E6 Org isolation probe.
PASS: denied as expected.
FAIL: exposure.
### 7.3 Gate Set F — operations readiness
F1 Error/latency baseline.
PASS: within thresholds.
FAIL: breach.
F2 On-call handoff.
PASS: on-call acknowledges release context.
FAIL: no acknowledgment.
F3 Internal release note.
PASS: note posted with ID and hashes.
FAIL: missing note.
F4 Evidence index publication.
PASS: index and links resolvable.
FAIL: missing/broken links.
F5 Rollback checkpoint.
PASS: rollback target remains valid post-release.
FAIL: uncertainty.
### 7.4 Post-release decision rule
All D/E/F PASS => STABLE.
Any D FAIL => PARITY FAILURE.
Any E FAIL => FUNCTIONAL FAILURE.
Any F FAIL => OPERATIONAL RISK.
PARITY FAILURE => rollback decision in <=10 minutes.
FUNCTIONAL FAILURE => incident declaration + mitigation path.
OPERATIONAL RISK => on-call escalation and timed remediation.

## 9) Failure handling when parity evidence is missing or mismatched
### 9.1 Missing required parity evidence before GO
Condition: required artifact absent at pre-release freeze.
Actions:
- Declare NO-GO immediately.
- Open `REL-PARITY-MISSING` incident.
- Assign owner in <=5 minutes.
- Regenerate from CI source only.
- Re-run full pre-release checklist.
### 9.2 Hash mismatch before GO
Condition: recomputed hash != recorded hash.
Actions:
- Declare NO-GO immediately.
- Freeze release branch.
- Disable release execution credentials pending triage.
- Run integrity triage checklist.
- Escalate to Rafael and Marcus.
- Require new Release ID after remediation.
### 9.3 Missing required evidence after deployment
Condition: post-release required artifact absent.
Actions:
- Classify PARITY FAILURE.
- Start 10-minute rollback decision timer.
- If artifact cannot be recovered and verified in timer window, rollback.
- Log incident and control gap.
### 9.4 Runtime parity mismatch after deployment
Condition: runtime SHA/Release ID/hash mismatch against approved pair.
Actions:
- Classify SEV-1 Release Integrity Incident.
- Pause new signing workflows if integrity uncertain.
- Execute rollback to last-known-good unless compliance explicitly approves hold.
- Notify incident channel at 0/10/resolution intervals.
- Complete RCA within 24 hours.
### 9.5 Required parity failure communications
0-minute template:
"Parity failure detected for <release-id>. Release frozen. Triage in progress. Next update in 10 minutes."
10-minute template:
"Parity failure scope: <scope>. Recovery path: <rollback|rebuild>. ETA: <time>."
Resolution template:
"Parity incident resolved for <release-id>. Final state: <rolled back|stabilized>. Evidence index updated."
### 9.6 Required corrective-action fields
Every parity incident record must include:
- Root cause.
- Control failure mode.
- Corrective action.
- Preventive control update.
- Owner.
- Due date.
- Verification date.
Incident cannot close without all fields.

## 10) Thursday-launch-to-routine ops transition plan
Objective:
Convert Thursday launch discipline into repeatable routine operations without reducing control strength.
### 10.1 Transition phases
T0: Thursday launch week.
T1: First two routine trains.
T2: Steady-state routine operations.
### 10.2 T0 controls (launch week)
- Full pre-release and post-release checklists mandatory.
- Live witness: Jonas + Cilla + Marcus.
- All six pre-release evidence receipts attached before GO.
- Day+1 parity spot-check mandatory.
- Same-day 20-minute post-release review mandatory.
### 10.3 T1 controls (first two routine trains)
- Keep full parity controls unchanged.
- QA witness remains mandatory.
- Compliance witness required when compliance-impacting changes exist.
- Day+1 spot-check required for first routine train only.
- Convert manual screenshot capture to automated capture where possible.
- Track checklist duration and first-pass failure rate.
### 10.4 T2 controls (steady-state)
- Immutable FE/BE pairing remains mandatory baseline.
- Binary gates remain mandatory baseline.
- Required approvals remain mandatory baseline.
- Manual steps may reduce only where automation proves equivalent evidence quality.
- Quarterly evidence repository audit remains mandatory.
### 10.5 Transition exit criteria
Exit T0 -> T1 requires:
- One Thursday release with zero parity failures.
- Complete required evidence bundle.
- Post-release state STABLE.
Exit T1 -> T2 requires:
- Two consecutive routine trains with zero parity failures.
- No missing required approvals.
- Full checklist completion without skipped gates.
- No unresolved P0/P1 release incidents.
### 10.6 Transition metrics
M1 Parity pass rate per release. Target: 100%.
M2 Required evidence completeness rate. Target: 100%.
M3 Pre-release first-pass rate. Target: >=95%.
M4 Post-release STABLE rate. Target: >=95%.
### 10.7 Routine operations handoff requirements
- Support escalations must include Release ID.
- On-call runbook must include release evidence lookup path.
- QA must link checklist JSONs in release thread.
- Compliance must retain read access to evidence index.
- Platform must test evidence-path write permissions weekly.

- Weekly release-record review by Release Controller.
- Monthly cross-functional control review (Platform + QA + Compliance).
- Quarterly retention and restore drill review.
Training:
- Release-capable engineers must complete control walkthrough before access grant.
