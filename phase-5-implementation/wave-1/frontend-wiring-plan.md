## Section 1: The `_generated/api.ts` Dependency

**This is the most important thing in this document. Read it before doing anything else.**

The file `convex/_generated/api.ts` is auto-generated by the Convex CLI when you run `npx convex dev`. It contains the fully-typed API reference object (`api`) that the frontend imports to call Convex queries and mutations.

**Until this file exists and is up-to-date, none of the wiring work below is possible.** TypeScript will throw "Module not found" or "Property does not exist" on every `api.*` call.

### What Chloe must run before wiring

```bash
# Terminal 1 — start the Convex watcher
# This deploys functions AND keeps _generated/api.ts fresh
npx convex dev

# Leave this running. Every change to convex/*.ts files triggers a re-deploy
# and regenerates _generated/api.ts automatically.

# Terminal 2 — Next.js dev server
pnpm --filter web dev
```

If you only need to generate the types (not start a watcher), run:
```bash
npx convex dev --once
# Generates _generated/api.ts and exits. Use this for CI or one-shot type generation.
```

### What the generated file provides

After running `npx convex dev`, the file at `convex/_generated/api.ts` exports the `api` object:

```typescript
// convex/_generated/api.ts (auto-generated — do not edit)
export const api: {
  workOrders: {
    createWorkOrder: FunctionReference<"mutation", "public", CreateWorkOrderArgs, WorkOrderId>,
    listWorkOrders: FunctionReference<"query", "public", ListWorkOrdersArgs, WorkOrderDoc[]>,
    getWorkOrder: FunctionReference<"query", "public", { workOrderId: Id<"workOrders"> }, WorkOrderDoc | null>,
    openWorkOrder: FunctionReference<"mutation", ...>,
    closeWorkOrder: FunctionReference<"mutation", ...>,
    voidWorkOrder: FunctionReference<"mutation", ...>,
    getCloseReadinessReport: FunctionReference<"query", ...>,
    // ...
  },
  taskCards: {
    createTaskCard: FunctionReference<"mutation", ...>,
    completeStep: FunctionReference<"mutation", ...>,
    signTaskCard: FunctionReference<"mutation", ...>,
    listTaskCardsForWorkOrder: FunctionReference<"query", ...>,
    // ...
  },
  // ... all other namespaces
} = /* implementation */;
```

The import path in frontend components:
```typescript
import { api } from "@/convex/_generated/api";
// (with Next.js @/ alias pointing to the repo root)
```

> **TypeScript error note:** Until `npx convex dev` has been run at least once in your checkout, `pnpm turbo typecheck` will fail with module not found errors. This is expected and documented in environment-setup.md §6 ("pnpm turbo typecheck fails after pulling — Fix: run `npx convex dev --once`").

## Section 3: Replacement Pattern — Step by Step

Every stub follows the same four-step replacement:

### Step 1 — Remove the stub import
```typescript
// DELETE THIS:
const api = {} as any;
```

### Step 2 — Add the real Convex import
```typescript
// ADD THIS at the top of the file with other imports:
import { api } from "@/convex/_generated/api";
import { useQuery, useMutation } from "convex/react";
// Only import what you use — don't import both if you only need one
```

### Step 3 — Replace the type-cast call
```typescript
// BEFORE (stub):
const workOrders = useQuery(api.workOrders.listWorkOrders, {} as any);

// AFTER (real):
const workOrders = useQuery(api.workOrders.listWorkOrders, {
  statuses: activeStatuses,
  // TypeScript will now tell you the exact args shape — don't guess
});
```

### Step 4 — Replace local type interfaces with generated types
```typescript
// BEFORE (stopgap local interface):
interface WorkOrderDoc {
  _id: string;
  workOrderNumber: string;
  status: string;
  // ...
}

// AFTER (generated type):
import type { Doc } from "@/convex/_generated/dataModel";
type WorkOrderDoc = Doc<"workOrders">;
// Or with denormalized fields from the query:
type WorkOrderWithDetails = Doc<"workOrders"> & {
  tailNumber: string;
  customerName: string;
  assignedTechNames: string[];
};
```

> **Golden rule:** If TypeScript isn't complaining, the types match. If TypeScript IS complaining after you replace the stub, it means there's a real mismatch between what the component expects and what the Convex function returns. Fix the TypeScript error — do not cast to `any` to silence it. Every `any` cast re-introduces the exact problem we're trying to solve.

### Test failures that Chloe must fix (per test-import-fix.md)

Jonas's document covers the Convex function test files (`workOrders.test.ts`, `taskCards.test.ts`). Those are not Chloe's files — Cilla and Devraj own those. However:

**`apps/web` Vitest tests** (if any component tests exist) may fail after wiring if:
- Components under test previously mocked the `api` object with `{} as any` in the mock
- After wiring, the real import is used and the test environment needs to be updated to mock `convex/react`

Standard pattern for mocking Convex in component tests:
```typescript
// In test files — mock the convex/react module
import { vi } from "vitest";
vi.mock("convex/react", () => ({
  useQuery: vi.fn(),
  useMutation: vi.fn(),
  useConvexAuth: vi.fn().mockReturnValue({ isAuthenticated: true, isLoading: false }),
}));

// Then in each test:
import { useQuery } from "convex/react";
(useQuery as ReturnType<typeof vi.fn>).mockReturnValue([/* mock data */]);
```

This pattern is cleaner than `{} as any` and doesn't require a running Convex instance in tests.
