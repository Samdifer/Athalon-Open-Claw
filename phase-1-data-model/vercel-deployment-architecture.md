> **[RM]** The `convex/` directory sits at the repo root rather than inside `apps/web/` — this is intentional. Convex functions are not "part of the web app"; they are a separate deployment target. Co-locating them with the web app in a monorepo is fine, but mixing them inside the Next.js app directory creates confusion about what runs where. Keep the boundary clean. The `packages/types` shared package is the bridge: both `apps/web` and `convex/` import from it, ensuring the domain model is never duplicated.

## 2. Environment Strategy

### 2.1 Three Environments

| Environment | Branch | Convex Project | Vercel Target | Clerk Instance | Purpose |
|---|---|---|---|---|---|
| **Development** | `feat/*`, `fix/*`, local | `athelon-dev-[username]` | Preview URL | Development instance | Individual feature work |
| **Staging** | `main` (pre-public) | `athelon-staging` | `staging.athelon.app` | Development instance | Integration, design review, QA |

**Phase 1 reality:** "Production" is provisioned and tested but not customer-facing. We build and deploy to it to ensure the pipeline works and secrets are correct. No real users.

### 2.3 Environment Variables — Naming Conventions

All env vars follow a strict naming pattern: `{SCOPE}_{SERVICE}_{DESCRIPTOR}`.


#### Per-Environment Variable Sets (Vercel)

Vercel has three variable scopes: Production, Preview, and Development. We use them as follows:

| Variable | Production | Preview | Development |
|---|---|---|---|
| `NEXT_PUBLIC_CLERK_PUBLISHABLE_KEY` | `pk_live_...` | `pk_test_...` | `pk_test_...` |
| `CLERK_SECRET_KEY` | `sk_live_...` | `sk_test_...` | `sk_test_...` |
| `NEXT_PUBLIC_CONVEX_URL` | `https://athelon-prod.convex.cloud` | `https://athelon-staging.convex.cloud` | (local dev: not set, uses `.env.local`) |
| `CONVEX_DEPLOY_KEY` | `prod:...` | `preview:...` | not set |

The Preview scope in Vercel maps to ALL preview deployments (all PRs). Preview deployments hit the **staging** Convex project — not a per-PR Convex instance.

> **[RM]** I want to flag a future improvement here: eventually we should have per-PR Convex dev branches to isolate schema experiments. Convex's branching feature supports this. For Phase 1, all preview deployments share staging Convex to reduce operational overhead. We'll revisit when we have more active concurrent feature work.

## 3. Preview Deployment Workflow

### 3.1 PR Lifecycle

```
Developer opens PR
    │
    ├─► GitHub Actions CI runs (lint, typecheck, test, security)
    │       └─► Results posted as PR status checks
    │
    ├─► Vercel automatically detects PR via GitHub integration
    │       └─► Builds and deploys preview to unique URL
    │       └─► Posts preview URL as PR comment
    │
    ├─► If convex/schema.ts or convex/*.ts changed:
    │       └─► CI posts "Convex function diff" comment on PR
    │           (what functions changed, whether schema migration needed)
    │
    └─► Reviewer checks:
            1. CI all green
            2. Preview URL loads correctly
            3. Design review (Chloe / Finn sign-off if UI change)
            4. Schema review (Rafael sign-off if schema change)
            └─► Merge gated on all checks passing
```

### 3.2 Vercel Preview Configuration

Vercel detects the GitHub integration automatically. Key configuration:

```
Vercel Project Settings:
  Git Integration: Connected to athelon/athelon repo
  Production Branch: main
  Preview Branches: all (every PR gets a preview)
  Ignored Build Step: (not set — build every push)
```

**Preview URL pattern:** `athelon-git-[branch-slug]-athelon-team.vercel.app`
These URLs are not indexed (Vercel sets `X-Robots-Tag: noindex` on preview deployments).

Preview deployments inherit the "Preview" environment variables from Vercel, which point to staging Convex and Clerk's test instance.

### 3.3 Convex Schema Change Detection

```yaml
# .github/workflows/ci.yml (schema change detection step)
- name: Detect Convex schema changes
  id: convex_diff
  run: |
    git diff origin/main...HEAD --name-only | grep -E '^convex/' && echo "changed=true" >> $GITHUB_OUTPUT || echo "changed=false" >> $GITHUB_OUTPUT

- name: Post Convex diff summary
  if: steps.convex_diff.outputs.changed == 'true'
  uses: marocchino/sticky-pull-request-comment@v2
  with:
    header: convex-diff
    message: |
      ## ⚠️ Convex Functions Changed
      This PR modifies files in `convex/`. 
      
      **Required:** Rafael Mendoza or Jonas Harker must review before merge if `schema.ts` changed.
      
      Changed files:
      ${{ steps.convex_diff.outputs.files }}
```

### 3.4 Design Review Flow

The staging Convex project is seeded with synthetic test data that covers the main entity types (aircraft, open work orders, signed task cards). I'll maintain the seed script so Chloe and Finn can always load the preview URL and see a realistic-looking app without needing real shop data.

### 4.1 Pipeline Overview

```yaml

name: CI

  pull_request:
    branches: [main]
  push:
    branches: [main]

  group: ${{ github.workflow }}-${{ github.ref }}
```

```typescript
// convex/workOrders.test.ts
import { convexTest } from "convex-test";
import { describe, it, expect } from "vitest";
import { api } from "./_generated/api";
import schema from "./schema";

describe("createWorkOrder", () => {
  it("rejects unauthenticated callers", async () => {
    const t = convexTest(schema);
    await expect(
      t.mutation(api.workOrders.createWorkOrder, {
        orgId: "k57..." as any,
        aircraftId: "k12..." as any,
        description: "100hr inspection",
      })
    ).rejects.toThrow("Unauthenticated");
  });

  it("rejects callers with viewer role", async () => {
    const t = convexTest(schema);
    const { user, org } = await t.run(async (ctx) => setupViewerUser(ctx));
    await expect(
      t.withIdentity({ subject: user.clerkId }).mutation(api.workOrders.createWorkOrder, {
        orgId: org._id,
        aircraftId: "k12..." as any,
        description: "100hr inspection",
      })
    ).rejects.toThrow("Insufficient role");
  });

  it("allows AMT or higher to create work orders", async () => {
    const t = convexTest(schema);
    const { user, org, aircraft } = await t.run(async (ctx) => setupAmtUser(ctx));
    const woId = await t.withIdentity({ subject: user.clerkId }).mutation(api.workOrders.createWorkOrder, {
      orgId: org._id,
      aircraftId: aircraft._id,
      description: "Annual inspection",
    });
    expect(woId).toBeDefined();
  });
});
```

> **[RM]** Every auth-guarded Convex mutation gets at minimum three tests: unauthenticated rejection, under-privileged role rejection, and success with minimum required role. This is not optional — it's the test baseline for any mutation that touches compliance-relevant data. I'll call this out in code review if it's missing.

### 5.1 Error Tracking — Sentry

Sentry is the primary error aggregation and alerting tool for both the Next.js frontend and Convex-originated errors.

**Frontend Sentry setup:**

```typescript

  tracesSampleRate: process.env.VERCEL_ENV === "production" ? 0.1 : 1.0,
  // Attach user context from Clerk session
  beforeSend(event) {
    // Strip PII from breadcrumbs before sending
    return event;
  },
});
```

```typescript
// apps/web/sentry.server.config.ts
import * as Sentry from "@sentry/nextjs";

```

```typescript
// apps/web/app/layout.tsx — Attach user context to Sentry
import * as Sentry from "@sentry/nextjs";
import { auth, currentUser } from "@clerk/nextjs/server";

export default async function RootLayout({ children }) {
  const { userId, orgId } = auth();
  if (userId) {
    Sentry.setUser({ id: userId });
    Sentry.setTag("org_id", orgId ?? "none");
  }
  return <html>{children}</html>;
}
```

**Convex error forwarding:** Convex does not natively integrate with Sentry. We handle this with a wrapper:

```typescript
// convex/lib/sentry.ts
export function withSentry<T>(
  label: string,
  fn: () => Promise<T>
): Promise<T> {
  return fn().catch((err) => {
    // In production Convex, log to Logtail (see 5.2) with structured error data
    console.error(JSON.stringify({
      level: "error",
      label,
      message: err.message,
      stack: err.stack,
    }));
    throw err; // Re-throw so Convex surfaces it to the caller
  });
}
```

Convex function errors surface to the frontend via Convex's error propagation. The frontend Sentry instance catches them as unhandled exceptions in `useQuery` / `useMutation` error paths.

### 5.2 Structured Logging — Logtail (Better Stack)

Logtail (Better Stack) is the centralized log aggregation service. Convex ships logs via the Logtail source integration.

**Why Logtail and not Datadog?** Cost. Datadog at Athelon's Phase 1 scale is overkill. Logtail is purpose-built for structured logs, has a generous free tier, and integrates directly with Vercel and Convex. We'll revisit Datadog at Series A.

**Convex logging convention:**

```typescript
// convex/lib/logger.ts
type LogLevel = "info" | "warn" | "error";
type LogContext = Record<string, string | number | boolean | null>;

export function log(level: LogLevel, message: string, ctx: LogContext = {}) {
  const entry = {
    level,
    message,
    timestamp: new Date().toISOString(),
    ...ctx,
  };
  // Convex runtime console output is captured by Logtail source integration
  console[level](JSON.stringify(entry));
}
```

All Convex functions log at start and completion for mutations that touch compliance data:

```typescript
// convex/workOrders.ts (example instrumented mutation)
export const createWorkOrder = mutation({
  handler: async (ctx, args) => {
    const { user, org } = await requireOrgContext(ctx);
    log("info", "work_order.create.start", {
      userId: user._id,
      orgId: org._id,
      aircraftId: args.aircraftId,
    });

    // ... business logic ...

    log("info", "work_order.create.complete", {
      userId: user._id,
      orgId: org._id,
      workOrderId: newId,
    });
    return newId;
  },
});
```

**Vercel log drain:** Configure Vercel to ship logs to Logtail:

```

  Sources: edge, serverless, static
```

### 5.4 Performance Monitoring — Vercel Analytics + Speed Insights

```typescript
// apps/web/app/layout.tsx
import { Analytics } from "@vercel/analytics/react";
import { SpeedInsights } from "@vercel/speed-insights/next";

export default function RootLayout({ children }) {
  return (
    <html>
      <body>
        {children}
        <Analytics />
        <SpeedInsights />
      </body>
    </html>
  );
}
```

Core Web Vitals targets:

| Metric | Target | Alert if |
|---|---|---|
| LCP (Largest Contentful Paint) | < 2.5s | > 3.0s (p75) |
| INP (Interaction to Next Paint) | < 200ms | > 500ms (p75) |
| CLS (Cumulative Layout Shift) | < 0.1 | > 0.15 (p75) |

Vercel Speed Insights reports these automatically. Alerts are manual review in Phase 1 (no automated perf alerts yet).

> **[RM]** CLS is worth calling out specifically. Aviation mechanics using this on tablets in poor lighting conditions with gloves on will notice layout shift more than a typical user will. If the page jumps while they're trying to tap something, they'll report it as a bug even if it's technically a "minor" shift. Keep CLS near zero, not just under the threshold.

### 5.5 Auth Failure Alerting

Authentication failures at scale can indicate a brute force attack or a misconfigured credential rotation. We alert on anomalies.

**Clerk webhook → Logtail → Alert rule:**

Clerk fires `session.ended` and can fire events for failed sign-in attempts. We configure a Logtail alert rule:

```
Alert: auth_failure_spike
  Query: level:error AND label:clerk.auth_failure
  Condition: count > 10 in 5 minutes
  Action: PagerDuty + Slack #security
```

This rule is defined in Logtail's alert configuration (not in code — it's a dashboard config). Document it in the infrastructure changelog.

## 6. Rollback Procedures

### 6.1 Vercel Rollback (Instant)

```bash
# Via CLI
vercel rollback [deployment-url-or-id] --scope athelon

# Via Vercel dashboard
# Deployments → Select previous deployment → Promote to Production
# Takes approximately 30 seconds.
```

No code change required. This is the first action on any production incident.

### 6.2 Convex Rollback (Requires Care)

Convex does not have a one-click schema rollback. If a schema migration breaks something:

1. **Identify last good schema commit:**
   ```bash
   git log --oneline convex/schema.ts
   git checkout <good-sha> -- convex/schema.ts
   ```

2. **Author and apply inverse migration:**
   - If you added a field: write a migration that removes it (or makes it optional)
   - If you removed a field: this is harder — data is gone. This is why breaking migrations require Rafael or Jonas approval before merge.

3. **Deploy the corrected schema:**
   ```bash
   CONVEX_DEPLOY_KEY=prod:... npx convex deploy
   ```

4. **Simultaneously rollback Vercel** to the deployment that matched the previous schema.

5. **Post-incident:** Document in the infrastructure changelog. Root cause analysis within 48 hours.

**RTO target:** 30 minutes for full restore to known good state.

### 6.3 Feature Flag Rollback

For lower-risk features, use the `NEXT_PUBLIC_FF_*` feature flag env vars. Setting a flag to `false` in Vercel environment variables and triggering a redeploy takes ~2 minutes. This is faster than a code rollback for UI-only regressions.
